# REPO_STRUCTURE.md — MSc Repository Structure Standard

This file defines the intended structure for the MSc repository. It describes the target state; the migration that gets us here is described in `operations/MIGRATION_PLAN.md`, and per-batch progress is in `operations/MIGRATION_LOG.md`.

## Top-level layout

```text
.github/        Workflows (deck export, cockpit build, link check)
capabilities/   Reusable skill cards, patterns, anti-patterns, case studies, specialisms
charter/        Slow-change policy and rule docs (this file lives here)
cockpit/        PWA source (HTML shells reading from operations/ and datasets/)
curriculum/     Assignments, courses, modules, thesis
datasets/       Source/reference data, including synthetic/
decks/          Slide-deck HTML source; .pptx generated by workflow
design/         Reusable design standards
docs/           Published PWA artifacts (GitHub Pages root)
notebooks/      Exploratory notebooks
operations/     Live registers, logs, queues, trackers
projects/       Active project work
prompts/        Reusable prompt templates
references/     Cited academic literature
reports/        Evidence reports
src/            Reusable library code
CLAUDE.md       Agent operating instructions
README.md       Repo landing
```

Only `README.md` and `CLAUDE.md` live at the root. Every other Markdown file lives in `charter/` or `operations/`.

## Folder responsibilities

### `charter/`

Slow-change policy. Files here change on the order of months and require PR review. Includes the twelve hard rules (`HARD_RULES.md`), persona mandates (`CODEMIKE.md`), workflow standards (`GITHUB_WORKFLOW.md`, `QA_CHECKLIST.md`), data and AI policy (`DATA_POLICY.md`, `RESPONSIBLE_AI.md`), tooling and budget rules (`TOOLING.md`, `BUDGET.md`), structural docs (this file, `DOMAIN_MAP.md`), and student-life orientation (`STUDENT_LIFE.md`, `SUPERVISION.md`, `orientation/`).

### `operations/`

Live registers and logs. Files here change on the order of days as work is logged. Includes the project log, roadmap, decision log, evidence and experiment registers, failure log, risk register, skill map, capability registry, dataset registry, library, portfolio, viva, transfer log, next actions, migration plan, and migration log.

Treat operations files as append-mostly. Editing an existing entry is allowed; deleting one requires a note in `DECISIONS.md`.

### `capabilities/`

Reusable skill cards plus supporting structure:

```text
capabilities/
├── <capability-name>/        skill cards
├── patterns/                 reusable methods
├── anti-patterns/            cautionary cases
├── case-studies/             applied examples
└── decision-science/         specialism content
```

When a capability becomes a viva subject, the defence deck goes in `decks/viva/`.

### `curriculum/`

Academic process and evidence, with assignments, courses, modules, and the thesis all under one parent. The course layout established for `DES-001` (`README.md`, `syllabus.md`, `weekly-plan.md`, `learning-outcomes.md`, `grading-policy.md`, `competency-map.md`, `learning-log.md`, plus `lectures/`, `readings/`, `labs/`, `quizzes/`, `assignments/`, `submissions/`, `feedback/`, `revisions/`, `viva/`, `portfolio/`) remains valid under `curriculum/courses/<course-code>/`.

### `cockpit/`

PWA source. Each cockpit view is an HTML shell that fetches data from `operations/*.json` or `datasets/*.csv` at runtime. No academic claim is hardcoded in cockpit HTML (HR-MSc-9). Build output ships to `docs/` for GitHub Pages.

### `decks/`

Slide-deck source. Decks are HTML (reveal.js or equivalent) that fetch their content from `operations/` and `datasets/` the same way cockpit views do. `.pptx` exports are produced by `.github/workflows/decks-export.yml` and offered as workflow artifacts, not committed.

### `datasets/`

CSV/JSON/reference datasets. Synthetic data lives in `datasets/synthetic/`. Each dataset is registered in `operations/DATASETS.md` with classification per `charter/DATA_POLICY.md`.

### `design/`

Reusable design knowledge: tokens, foundations, checklists. Course-specific design submissions live under `curriculum/`, not here.

### `docs/`

Published HTML for GitHub Pages. Treat as build output for `cockpit/` and the published deck index. Do not author long-form content here.

### `notebooks/`, `src/`, `projects/`, `prompts/`, `references/`, `reports/`

Conventional locations. Each artifact must belong to a module, project, experiment, or capability (HR-MSc-12).

## Naming conventions

Use stable prefixes:

| Type | Example |
|---|---|
| Topic note | `topic-02-what-is-ui-design.md` |
| Lecture | `lecture-02-what-is-ui-design.md` |
| Reading pack | `topic-02-what-is-ui-design-reading-pack.md` |
| Lab | `lab-02-ui-component-inventory.md` |
| Lab result | `lab-02-ui-component-inventory-results.md` |
| Grade report | `DES-001-grade-report-v1.md` |
| Revision plan | `DES-001-revision-plan-v1.md` |
| Capability card | `cap-recommendation-scoring.md` |
| Pattern | `pat-baseline-before-model.md` |
| Anti-pattern | `anti-dashboard-without-decision.md` |
| Migration batch | `batch-03-move-operations.md` |

## Anti-sprawl rules

1. Avoid one-off folders unless they match the course or capability pattern.
2. Do not duplicate the same evidence in multiple places; link to the canonical source.
3. Use a course submission file as the index for evidence, not a copy of every artifact.
4. Prefer small source files plus generated/rendered artifacts. HTML, `.pptx`, and built PWA bundles are generated, not authored.
5. Move obsolete or superseded files to an archive only after they are clearly replaced.
6. Only `README.md` and `CLAUDE.md` live at the root.

## Migration state

This file describes the target state. The repository is in transition; until all batches merge, legacy locations and new locations may temporarily coexist. Per-batch progress is recorded in `operations/MIGRATION_LOG.md`.
