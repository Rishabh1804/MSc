## 3. Criterion Details

### 3.1 Multi-Source Coverage — 15 marks

High score evidence:

- each topic uses at least 3 sources
- source types are varied
- primary/high-authority source included where possible
- practical/applied source included
- cross-check or alternative framing source included
- source selection rationale is clear

Low score evidence:

- one-source summaries
- source list is decorative but not used
- only vendor/product-library sources used
- no source classification

### 3.2 Notes Quality and Topic Summaries — 15 marks

High score evidence:

- concise but useful notes per topic
- key principles are accurately captured
- notes avoid vague design language
- topic definitions are practical and understandable
- notes distinguish principle from implementation

Low score evidence:

- generic summaries
- copied-sounding fragments without synthesis
- no distinction between UI, UX, usability, visual design, and implementation
- shallow notes that cannot guide actual design decisions

### 3.3 Source Comparison and Bias Awareness — 15 marks

High score evidence:

- explains where sources agree
- explains where sources differ
- identifies product/vendor bias where relevant
- identifies omissions such as accessibility, data-heavy UI, mobile, engineering constraints, or research evidence
- avoids treating one source as final truth

Low score evidence:

- no comparison
- all sources treated as equivalent
- ignores bias or context
- no mention of disagreements or limitations

### 3.4 Application to CodeMike / Browser Context — 20 marks

High score evidence:

- each topic is translated into destination master browser implications
- recommendations connect to actual user tasks
- design choices improve review, filtering, inspection, trust, or safety
- clearly distinguishes structural validation from Planner-readiness
- produces implementation implications for v1.1

Low score evidence:

- notes remain theoretical
- no clear link to the current HTML artifact
- design ideas are decorative rather than functional
- warning/status/data-trust requirements are weak

### 3.5 Further Reading and Learning Path — 10 marks

High score evidence:

- every topic has further reading suggestions
- suggestions are relevant and purposeful
- learning path is sequenced logically
- further reading includes practical and conceptual next steps

Low score evidence:

- no further reading
- generic suggestions
- links do not deepen the topic
- no relationship to current project needs

### 3.6 HTML Usability and Clarity — 10 marks

High score evidence:

- dashboard is easy to navigate
- topic status is visible
- source lists and notes are readable
- responsive layout works acceptably
- assignment identity is clear
- the artifact can actually be reviewed/marked

Low score evidence:

- cluttered or confusing layout
- notes hard to scan
- poor mobile behaviour
- assignment purpose unclear
- visual polish interferes with reading

### 3.7 Checklist / Actionability — 10 marks

High score evidence:

- outputs can update `master-browser-design-checklist.md`
- recommendations are actionable
- implementation implications are specific
- audit criteria are usable before redesign

Low score evidence:

- no checklist impact
- vague recommendations
- no decision criteria
- cannot guide browser v1.1

### 3.8 Academic Discipline / Versioning — 5 marks

High score evidence:

- assignment brief exists
- rubric exists
- submission file exists at review stage
- review file exists after marking
- status and version are clear
- resubmission/benchmark status is recorded

Low score evidence:

- no submission/review trail
- no versioning
- unclear status
- no benchmark/resubmission rule

## 4. Automatic Caps
