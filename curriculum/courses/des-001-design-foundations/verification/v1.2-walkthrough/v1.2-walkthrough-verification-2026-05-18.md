## Implementation notes per acceptance criterion

- **U-CONF-1**: The modal title dynamically reflects the exact selected count ("Confirm: promote N records to Planner"). The list shows up to 25 records by name + ID; for N > 25 the overflow is named (`+ N more`) per Lab 04 Loop 1 §6 ASCII layout `(scroll to see all)`.
- **U-CONF-2**: Cancel is wired to button click + Esc keydown + overlay click. No confirmation-of-cancel; the cancel path is single-interaction by design per Topic 4 §3 asymmetric-cost framing (the cost of cancelling is small; the cost of a wrong promotion is large).
- **U-CONF-3**: `aria-modal="true"` distinguishes this from the existing drawer (which is `aria-modal="false"` per Topic 2 §3.5). The modal body text starts with a `<p class="lead">` element naming the consequence; the lead text is announced as the dialog description.
- **U-CONF-4**: Focus trap is implemented in `handlePromoteModalKey()` — Tab and Shift+Tab cycle among the two focusable elements in the modal footer (Cancel + Confirm). Focus restoration uses `state.promoteOpener` captured at open-time; on close, that element receives focus back. The opener is captured per-call (defaults to `document.activeElement` if not explicitly passed).

## Lab 04 §State machine: 3 modal states honoured

| State | Implementation |
|---|---|
| `default` | Modal opens with Cancel focused; body lists records; both buttons enabled |
| `busy` | Confirm button disabled (`disabled = true`) immediately after click — prevents double-confirm |
| `done` | Modal closes; toast shows "N records promoted to Planner"; selection cleared; underlying rows reflect new state (`planner_use_status = planner_ready`); trust badges re-render via `applyFilters()` |

(There is no `error` state in v1.2 because there is no real Planner backend — the promotion is in-session only. A real backend would add an error path that keeps the modal open with retry + report-issue actions per Lab 04 §State machine.)

## What this verification does NOT cover

- **Real Planner backend** — v1.2 simulates promotion in browser memory only. The HTML state change (`planner_use_status = planner_ready`) is session-scoped; a real backend integration is a separate scope item.
- **N > 25 record overflow path** — the test uses 3 records (well within the inline-display threshold). The overflow `+ N more` rendering is implementation-tested by the code but not walkthrough-tested with N > 25. Future Audit 2 should cover this.
- **Screen-reader actual behaviour** — the walkthrough confirms ARIA attributes are *present*; an actual screen-reader user (Sponsor Reviewer accessibility-need persona, per Lab 04 §2 Perspective 3) would verify the announcements are *useful*. Queued for first Sponsor Reviewer session.
- **Touch / mobile interaction** — desktop-shaped tests only. Mobile checkbox tap targets, modal sizing on narrow viewports — deferred per Lab 05 inclusion-lens findings.
- **Concurrent promotions / rate-limiting** — no real backend, so this is moot for v1.2; v2.x would test under load.

These limitations are named per Topic 5's HCD discipline. The feature is *HCD-substantial* per Lab 04 Loop 1's §State machine + the four U-CONF criteria; the limitations are *queued* (not silenced).
