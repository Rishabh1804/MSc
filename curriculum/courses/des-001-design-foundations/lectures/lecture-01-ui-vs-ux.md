# Lecture 01 — UI vs UX

## Learning objectives

By the end of this lecture, CodeMike should be able to:

- distinguish UI from UX
- explain why UI quality is not equal to UX quality
- identify UI elements in the Destination Master Browser
- map UI elements to reviewer tasks
- identify data-trust as a UX concern

## Core distinction

UI is the interface surface: visible controls, layout, labels, components, visual hierarchy, states, and interaction feedback.

UX is the broader user experience: the task journey, user goal, trust level, accessibility, recovery from errors, decision confidence, and overall usefulness.

## CodeMike interpretation

For the Destination Master Browser, UI asks:

```text
What does the reviewer see and interact with?
```

UX asks:

```text
Can the reviewer safely inspect, filter, compare, verify, and avoid over-trusting unverified records?
```

## Common misconception

A polished interface is not automatically a good user experience. A dashboard can look impressive but still fail if users cannot understand whether the data is verified or Planner-ready.

## Application to browser redesign

Every proposed UI change must answer:

1. Which reviewer task does this improve?
2. Which UI element or state delivers it?
3. Which data-trust or workflow risk does it reduce?

## Discussion questions

1. Which current browser elements are UI-only?
2. Which current browser elements directly support UX tasks?
3. Where can visual polish accidentally create false trust?
4. How should uncertainty be represented in a data-review interface?

## Key takeaway

UI is the mechanism. UX is the quality of the journey and decision outcome. In this project, trustworthy review is the UX goal.
