# MIGRATION_PLAN.md — MSc Repository Migration

Frozen target state for the repository restructure. This file does not change as batches execute; the log of what has actually happened lives in `MIGRATION_LOG.md`.

## Why

The root of the repo currently holds 32 Markdown files and 22 directories. New contributors and review tools cannot tell at a glance which files are slow-changing policy and which are live registers that update weekly. Capability artifacts, curriculum artifacts, and trackers are intermixed.

The migration sorts root content into four named buckets — charter (policy), operations (live registers), capabilities (reusable skill cards), curriculum (academic work) — and adds a cockpit PWA and a deck pipeline that read from those buckets without duplicating their content.

## Slide-deck pipeline

Slide decks follow HR-MSc-9: HTML source, data fetched from operations and datasets at runtime.

```text
decks/
├── curriculum/<module-slug>/
│   ├── slides.html       reveal.js source; fetches from operations/*.json
│   ├── notes.md          speaker notes
│   └── README.md         deck charter (intent, audience, evidence links)
├── viva/<capability-slug>/
│   └── slides.html       defence rehearsal
├── transfer/<transfer-slug>/
│   └── slides.html       capability-transfer pitch
└── shared/
    ├── reveal-theme.css  cockpit design tokens applied to slides
    └── deck-data-fetch.js fetch + escape helpers
```

`.pptx` export is generated by `.github/workflows/decks-export.yml` on push to `decks/**`. The workflow renders each deck via headless Chromium screenshots and assembles a `.pptx` with `python-pptx`. Output is uploaded as a workflow artifact, not committed.

## Idempotency

Each batch is a no-op if re-run on the target state. Move batches use `git mv`. Add batches check for existing target files and skip rather than overwrite. The migration log records the SHA of the merge commit for each batch.

## When this plan changes

Material changes (new batch, removed batch, retargeted file) require updating this file in a separate commit before the affected batch runs. The change is noted in `MIGRATION_LOG.md` under a `Plan revision` entry.
